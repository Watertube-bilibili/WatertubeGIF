# WatertubeGIF

WatertubeGIF 是一个基于 HarmonyOS ArkTS 的视频转 GIF 应用。它面向手机和平板，提供从视频选择、转换参数配置、GIF 生成、结果预览、保存到相册到历史记录管理的一整套流程。

项目仍在迭代中。本文档按当前代码状态说明功能、优点、构建方式和仍未修复的问题。

## 功能概览

- 从系统相册选择单个视频。
- 转换前可调整输出宽度、帧率、截取时长、循环播放和倒放。
- 提供三档输出质量预设：超高质量、中等质量、微信质量。
- 读取视频旋转元数据并智能计算最长边，避免竖屏视频生成宽体 GIF。
- 转换过程中展示进度。
- 生成后进入预览页，可保存 GIF 到系统相册。
- 首页显示最近转换历史，并支持清空历史记录。
- 设置页可配置默认质量、宽度、帧率、后台长任务和实况窗。
- 新增“水管实验室”：参数扭蛋、每日 GIF 黄历和历史抽卡。
- 可开启“社死保护”，避免在历史记录中显示原视频文件名。
- 支持开发者弹窗可设置七天内不再提醒。
- 接入后台连续任务和 Live View Kit，在状态栏胶囊、锁屏和通知中心展示 GIF 导出进度。

## 项目优点

### 1. 转换流程完整

应用不是选择视频后直接开转，而是先进入参数页。用户可以在转换前确认画面宽度、帧率、时长、循环和倒放设置，流程更可控。

核心页面：

- `Index.ets`：首页、历史记录、视频选择入口。
- `ConvertPage.ets`：转换参数和转换进度。
- `PreviewPage.ets`：结果预览和保存到相册。
- `SettingsPage.ets`：默认输出、后台转换、实况窗和界面材质状态。
- `LabPage.ets`：随机参数生成、每日提示和随机历史访问。

### 2. 输出比例处理更稳

转换核心会读取 `videoWidth`、`videoHeight` 和 `videoOrientation`。90°/270° 旋转的素材会先交换显示宽高；智能尺寸开启时，用户选定的数值是画面最长边，横屏和竖屏都按原比例计算另一边。算法不再单独抬高或压低高度，避免人为拉伸。

### 3. 长视频改为 FFmpeg 流式处理

项目允许处理最长两小时的视频片段。转换时不再把所有 `PixelMap` 帧保存在 ArkTS 数组中，而是由 FFmpeg 依次完成解码、抽帧、缩放和 GIF 写入。

实际帧数由用户选择的时长和帧率决定，不再受旧内核 180/360/720 帧上限限制。长视频仍会产生非常大的 GIF，并受到设备性能、剩余存储空间和系统后台策略限制。

### 4. 转换核心已做速度优化

当前核心优化点：

- 按华为“视频截取转换为 GIF 动图”行业示例接入 `@ohos/mp4parser`。
- 使用 `MP4Parser.ffmpegCmd()` 调用 FFmpeg 原生媒体链路。
- 移除 ArkTS 层逐帧读取、像素复制、调色板构建和 LZW 压缩。
- 相册媒体 URI 会先复制到应用沙箱，再交给 FFmpeg，兼容真实图库资源。
- FFmpeg 使用 Lanczos 缩放，并直接输出目标 GIF 文件。

### 5. 安装包体积优化

`@ohos/mp4parser` 默认同时携带 ARM64、32 位 ARM 和 x86_64 三套 FFmpeg 动态库，这是旧包体积超过 69 MB 的主要原因。当前发布包通过 Hvigor `abiFilters` 和依赖原生库过滤，仅保留 `arm64-v8a`：

- HAP 从约 69.1 MB 降至约 22.2 MB，减少约 67.9%。
- GIF 转换能力和 FFmpeg 功能保持不变。
- 发布包支持现代 HarmonyOS ARM64 真机，不再支持 x86_64 模拟器和 32 位 ARM 设备。

### 6. 三档质量预设

当前质量预设会真实影响转换参数和编码策略：

- 超高质量：使用更高宽度和较高帧率，优先保留画面细节。
- 中等质量：使用默认宽度和帧率，兼顾速度、体积和观感。
- 微信质量：限制宽度不超过 360、帧率不超过 10，优先控制体积，同时保持画面可辨识。

### 7. 使用华为官方行业示例链路

转换架构参考华为开发者文档“视频截取转换为 GIF 动图”，使用官方示例指定的 `@ohos/mp4parser` 第三方库调用 FFmpeg。FFmpeg 负责：

- 视频解码和按帧率取样。
- 按源视频比例缩放画面。
- GIF 颜色转换和压缩。
- 倒放滤镜。
- 单次播放或无限循环标记。

项目不再维护自研 GIF 调色板和 LZW 实现。

### 8. 减少颜色格式转换

视频帧的像素格式转换由 FFmpeg 原生媒体库完成，不再由应用读取 RGBA/BGRA 字节并手工交换通道，可减少额外复制和人为通道处理带来的偏色风险。

### 9. UI 已具备应用骨架

当前界面已包含：

- 首页。
- 可见的 GIF 标识。
- 最近记录列表。
- 空历史状态。
- 清空历史入口。
- 参数设置页。
- 转换进度页。
- 结果预览页。
- 保存到相册入口。
- 支持开发者弹窗。

### 10. 水管实验室

首页烧瓶按钮和设置页都可以进入实验室。这里的功能虽然带有随机性，但不会绕过正常转换流程：

- 参数扭蛋会在可用范围内随机组合质量、宽度、帧率、时长和倒放状态，选中视频后仍会进入参数页确认。
- 每日 GIF 黄历根据日期生成固定提示，同一天内不会因为刷新页面而变化。
- 历史抽卡会从现有转换记录中随机打开一项，不会修改或删除历史文件。
- 社死保护使用 Preferences 持久化；启用时会立即匿名化已有历史，新任务落库前也会再次检查设置。

## 未修复问题和已知风险

### 1. 超高质量仍受 GIF 格式限制

超高质量会尽量降低色差，但 GIF 格式本身只有 256 色，不可能做到和原视频完全一致。复杂渐变、暗部、肤色和游戏画面仍可能出现轻微色带或颜色损失。

### 2. 长视频可能生成极大文件

新内核按所选帧率持续生成，不再主动限制最大帧数。两小时、较高分辨率或较高帧率的 GIF 可能占用数百 MB 甚至更多空间，转换耗时也会明显增加。

转换是否成功仍取决于设备性能和剩余存储空间。

### 3. GIF 色彩质量仍有上限

GIF 格式本身只有 256 色。当前版本的颜色转换由 `@ohos/mp4parser` 内置的 FFmpeg 实现。

可能表现为：

- 渐变区域出现色带。
- 肤色、暗部或游戏画面出现轻微偏色。
- 多场景视频共用一个调色板时局部颜色损失明显。

复杂渐变、暗部和肤色仍无法达到原视频色深。

### 4. 取消转换不是即时硬中断

应用会在调用 FFmpeg 前检查取消状态，但 `@ohos/mp4parser` 当前没有提供中止 `ffmpegCmd()` 的接口。任务进入原生编码阶段后，取消不会立即生效。

后续需要等待依赖库提供 FFmpeg 任务中止能力，或接入支持取消的原生封装。

### 5. 后台任务和实况窗

GIF 转换属于 Live View Kit 允许的 `PROGRESS` 场景，即音视频编辑导出资源进度。当前实现包含：

- 用户主动开始转换时创建本地实况窗。
- 使用进度可视化卡片和进度胶囊展示状态。
- 读取 FFmpeg `-progress` 输出的已处理时间，换算并展示真实编码进度。
- 每次更新递增 `sequence`，避免更新被系统拒绝。
- 点击实况窗返回应用。
- 转换成功、失败时分别结束实况窗并保留短暂结果状态。
- 新任务启动前清理同 ID 的异常残留实况窗。
- 应用进入后台后使用连续任务维持 FFmpeg 转换。

正式展示仍依赖 AppGallery Connect 实况窗正式权益，以及用户在系统设置中开启本应用的实况窗开关。未取得权益、设备不支持或用户关闭开关时，应用会跳过实况窗并继续转换。

后台转换在 API 22 使用 `MODE_SPECIAL_SCENARIO_PROCESSING` 与 `SUBMODE_MEDIA_PROCESS_NORMAL_NOTIFICATION`，并由用户主动授权媒体处理长任务。`module.json5` 同时声明 `specialScenarioProcessing` 和兼容用的 `taskKeeping`。任务结束后会立即停止。

实况窗设置页提供真实连接测试，可区分系统开关未开启与 AppGallery Connect 权益未开通。实况窗权益绑定包名和正式签名，不能通过本地开关或清单文件伪造。

### 6. 转换偏好设置

首页齿轮按钮可进入设置页。设置使用 Preferences 持久化，新建转换任务会自动带入：

- 默认质量预设。
- 默认输出宽度。
- 默认帧率。
- 是否启用横竖屏智能尺寸。
- 是否申请媒体处理后台长时任务。
- 是否创建本地实况窗。
- 是否匿名化新旧历史记录名称。

### 7. 沉浸光感兼容状态

HarmonyOS 官方沉浸光感 `ImmersiveMaterial` 从 API 26 开始提供，并要求项目的目标 API 不低于 26。当前构建环境为 DevEco Studio 6.0.2、HarmonyOS SDK API 22，因此设置页使用现有系统模糊材质作为兼容降级，并明确显示沉浸光感不可用，不会把普通模糊效果冒充为新材质。

升级到 DevEco Studio 7 和 API 26 SDK 后，才可以在 `module.json5` 中启用 `ohos.arkui.UIMaterial.state` 并接入 `uiMaterial.ImmersiveMaterial`。

### 8. 历史记录只保存应用内文件路径

历史记录保存的是应用内部生成文件的位置。如果应用数据被清理，历史项可能还存在，但实际 GIF 文件已经不可用。

后续应在打开历史项时检查文件是否存在，并自动清理失效记录。

### 9. 保存到相册依赖系统权限

保存 GIF 需要写入图片/视频权限。用户拒绝权限或系统媒体库行为变化时，保存可能失败。

### 10. 图标组件仍是兼容实现

为了兼容当前 SDK，图标组件使用文本符号兜底，而不是全部使用系统 Symbol。首页主标识已改成稳定可见的 GIF 标识，但部分小图标仍可继续优化。

### 11. 发布包仍需要正式签名

自动构建产物仍是未签名包。正式上架前需要使用发布证书重新签名，并完成应用市场资料配置。

## 构建

推荐使用 DevEco Studio 或项目内 hvigor 构建。

项目依赖 `@ohos/mp4parser 2.0.8`，需要 API 22、HarmonyOS 6.0.2 SDK 和 DevEco Studio 6.0 或更高版本。当前 HAP 仅打包 ARM64 FFmpeg 原生库。

常用任务：

```powershell
hvigorw assembleHap --analyze=false --parallel --incremental
hvigorw assembleApp --analyze=false --parallel --incremental
```

常见输出：

- HAP：`entry-default-unsigned.hap`
- APP：`WaterTubeGIF2-default-unsigned.app`
