# WatertubeGIF

WatertubeGIF 是一个基于 HarmonyOS ArkTS 的视频转 GIF 应用。它面向手机和平板，提供从视频选择、转换参数配置、GIF 生成、结果预览、保存到相册到历史记录管理的一整套流程。

项目仍在迭代中。本文档按当前代码状态说明功能、优点、构建方式和仍未修复的问题。

## 功能概览

- 从系统相册选择单个视频。
- 转换前可调整输出宽度、帧率、截取时长、循环播放和倒放。
- 提供三档输出质量预设：超高质量、中等质量、微信质量。
- 按源视频比例计算输出尺寸，避免画面被压扁或只生成局部条状画面。
- 转换过程中展示进度。
- 生成后进入预览页，可保存 GIF 到系统相册。
- 首页显示最近转换历史，并支持清空历史记录。
- 支持开发者弹窗可设置七天内不再提醒。
- 尝试接入后台任务和 LiveViewKit，用于长任务进度展示。

## 项目优点

### 1. 转换流程完整

应用不是选择视频后直接开转，而是先进入参数页。用户可以在转换前确认画面宽度、帧率、时长、循环和倒放设置，流程更可控。

核心页面：

- `Index.ets`：首页、历史记录、视频选择入口。
- `ConvertPage.ets`：转换参数和转换进度。
- `PreviewPage.ets`：结果预览和保存到相册。

### 2. 输出比例处理更稳

转换核心会先读取视频首帧信息，再按源视频宽高比计算输出高度。输出宽高会限制在合理范围内并修正为偶数，减少视频比例异常导致的渲染问题。

### 3. 长视频不会直接打爆内存

项目允许处理长视频输入，但不会按完整帧率把所有帧都塞进内存。转换器会综合以下限制计算实际抽帧数：

- 用户选择的时长和帧率。
- 源视频真实时长。
- 最大帧数限制。
- 当前输出分辨率下的内存预算。

这套策略牺牲一部分长视频流畅度，换取移动设备上的稳定性。

### 4. 转换核心已做速度优化

当前核心优化点：

- 移除抽帧循环中的无效帧差采样，减少每帧额外 CPU 开销。
- UI 进度只在百分比变化时刷新，减少不必要的状态更新。
- LiveView 进度改为分段更新，避免每一帧都调用系统进度接口。
- GIF 调色板索引从全量预计算改为按实际出现颜色懒加载缓存，降低小图和低色彩场景下的编码启动开销。

### 5. 三档质量预设

当前质量预设会真实影响转换参数和编码策略：

- 超高质量：使用更高宽度、较高帧率、更高帧数上限、完整调色板采样和轻量抖动，优先降低色差。
- 中等质量：使用默认宽度和帧率，兼顾速度、体积和观感。
- 微信质量：降低宽度、帧率、帧数上限和采样预算，优先控制体积，同时保持画面可辨识。

### 6. GIF 编码逻辑可控

项目内置 GIF 编码器，负责：

- 调色板构建。
- 像素颜色映射。
- GIF 文件头和循环扩展块写入。
- 图像帧数据写入。
- LZW 数据输出。

不依赖额外 GIF 编码库，后续可以直接继续优化质量、速度和文件体积。

### 7. PixelMap 颜色格式有兼容处理

读取视频帧时会尽量转换为 `RGBA_8888`，并对 `BGRA_8888` 做通道兜底处理，减少红蓝通道错位导致的偏色问题。

### 8. UI 已具备应用骨架

当前界面已包含：

- 首页。
- 可见的 GIF 标识。
- 最近记录列表。
- 空历史状态。
- 清空历史入口。
- 参数设置页。
- 转换进度页。
- 结果预览页。
- 保存到相册入口。
- 支持开发者弹窗。

## 未修复问题和已知风险

### 1. 超高质量仍受 GIF 格式限制

超高质量会尽量降低色差，但 GIF 格式本身只有 256 色，不可能做到和原视频完全一致。复杂渐变、暗部、肤色和游戏画面仍可能出现轻微色带或颜色损失。

### 2. 长视频仍会稀疏抽帧

应用支持长视频输入，但当前不会完整保留长视频的所有帧。最大帧数限制仍然存在，所以长视频输出的 GIF 可能明显跳帧。

这是当前版本为了控制内存和文件体积做出的取舍。

### 3. GIF 色彩质量仍有上限

GIF 格式本身只有 256 色。当前编码器使用全局调色板和最近色映射，没有实现高质量抖动算法。

可能表现为：

- 渐变区域出现色带。
- 肤色、暗部或游戏画面出现轻微偏色。
- 多场景视频共用一个调色板时局部颜色损失明显。

后续可优化方向：

- 加入 Floyd-Steinberg 抖动。
- 使用更好的调色板量化算法。
- 为复杂场景支持局部调色板。

### 4. LZW 编码仍偏保守

当前 LZW 输出逻辑优先保证结构简单和可靠，还不是高压缩率实现。

影响：

- 输出 GIF 体积可能偏大。
- 文件写入和分享成本更高。

### 5. 取消转换不是即时硬中断

取消逻辑主要在抽帧循环中检查状态。如果任务已经进入编码阶段，取消可能不会立即生效。

后续需要把编码阶段也拆成可检查取消状态的分段任务。

### 6. 后台任务和实况窗存在系统兼容风险

代码已尝试接入后台任务和 LiveViewKit，但这些能力依赖系统版本、设备能力和权限状态。某些设备上可能无法启动实况窗或后台进度。

当前策略是捕获失败并继续执行转换，避免因为实况窗不可用导致转换本身失败。

### 7. 历史记录只保存应用内文件路径

历史记录保存的是应用内部生成文件的位置。如果应用数据被清理，历史项可能还存在，但实际 GIF 文件已经不可用。

后续应在打开历史项时检查文件是否存在，并自动清理失效记录。

### 8. 保存到相册依赖系统权限

保存 GIF 需要写入图片/视频权限。用户拒绝权限或系统媒体库行为变化时，保存可能失败。

### 9. 图标组件仍是兼容实现

为了兼容当前 SDK，图标组件使用文本符号兜底，而不是全部使用系统 Symbol。首页主标识已改成稳定可见的 GIF 标识，但部分小图标仍可继续优化。

### 10. 发布包仍需要正式签名

自动构建产物仍是未签名包。正式上架前需要使用发布证书重新签名，并完成应用市场资料配置。

## 构建

推荐使用 DevEco Studio 或项目内 hvigor 构建。

常用任务：

```powershell
hvigorw assembleHap --analyze=false --parallel --incremental
hvigorw assembleApp --analyze=false --parallel --incremental
```

常见输出：

- HAP：`entry-default-unsigned.hap`
- APP：`WaterTubeGIF2-default-unsigned.app`

## 发布说明

GitHub Release 通常上传 HAP 包；如果需要上传 APP 包，由于平台对 `.app` 扩展名有限制，应先打包为 zip 再上传。
